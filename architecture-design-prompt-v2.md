## Skill 定位

将 PRD(支持5W+字符) 和技术调研转化为架构设计文档，支持首次生成和增量更新。

## 执行流程

### 阶段 1：进入 Plan 模式
1. 使用 `EnterPlanMode` 工具进入计划模式
2. 读取输入文档，分析文档结构
3. 读取所有图片和文档引用

### 阶段 2：业务域识别与确认
1. 从 PRD 中识别所有业务域
2. 分析业务域之间的关联度和数据耦合度
3. **使用 `AskUserQuestion` 确认**：
   - 识别出的业务域是否完整？是否有遗漏？
   - 业务域的边界划分是否合理？
   - 提供 2-3 种业务域划分方案供选择，说明各方案利弊

### 阶段 3：微服务划分与确认
1. 基于确认的业务域，提出微服务聚合方案
2. 说明每个微服务聚合哪些业务域，以及聚合的理由
3. **使用 `AskUserQuestion` 确认**：
   - 微服务的聚合粒度是否合适？
   - 是否需要进一步拆分或合并？
   - 提供多种聚合方案（粗粒度 vs 细粒度），分析性能、复杂度、团队协作等维度

### 阶段 4：技术栈选型与确认
1. 从 research.md 中提取技术栈候选项
2. 结合业务特点，分析各技术栈的适用性
3. **使用 `AskUserQuestion` 确认**：
   - 推荐的技术栈是否符合团队技术储备？
   - 是否需要调整或替换某些技术选型？
   - 对于关键技术（如数据库、消息队列），提供 2-3 个选项及对比

### 阶段 5：核心流程设计与确认
1. 识别系统的核心业务流程（如下单流程、支付流程）
2. 设计流程图（Mermaid 格式）
3. **使用 `AskUserQuestion` 确认**：
   - 核心流程是否覆盖主要业务场景？
   - 流程中的关键节点（如异步处理、补偿机制）是否合理？
   - 是否有遗漏的异常处理路径？

### 阶段 6：生成文档
1. 使用 `ExitPlanMode` 退出计划模式
2. 基于所有确认的内容，生成完整的架构设计文档
3. 确认文档生成完成

## 文档结构要点

**必须包含**：
- YAML frontmatter（version、status、modules、change_log）
- 微服务/模块划分
- 技术栈选型
- 核心流程图（Mermaid）
- 模块清单（格式：`module-name: 描述`）

**不要包含**：
- 具体 API 设计
- 数据库表结构
- 高可用/容灾方案

## 关键设计点

### 模块清单格式

**微服务划分原则**：
- **一个微服务聚合多个相关业务域** - 不要一个域就拆一个微服务
- 按业务关联度和数据耦合度聚合，例如：用户域+权限域+组织域 → 用户中心服务
- 避免过度拆分导致分布式事务和网络开销
- 服务之间通过 API 调用或消息队列通信

**工作项划分原则**：
- 一个工作项对应一个业务域（不是微服务）
- 如果业务域过大，可拆分成多个工作项
- 工作项是技术设计和开发的最小单元

```markdown
## 后端微服务划分（示例）
- user-center-service: 用户中心服务，聚合用户域、权限域、组织域
- order-service: 订单服务，聚合订单域、支付域、履约域
- product-service: 商品服务，聚合商品域、库存域、类目域

## 工作项清单（按业务域拆分）
- user-domain: 用户域，负责用户注册、登录、个人信息管理
- permission-domain: 权限域，负责角色、权限、访问控制
- organization-domain: 组织域，负责部门、团队、层级管理
- order-domain: 订单域，负责订单创建、状态流转
- payment-domain: 支付域，负责支付渠道、支付回调
- fulfillment-domain: 履约域，负责发货、物流跟踪
```

**格式要求**（供 technical-design skill 解析）：
- 微服务格式：`service-name: 服务描述，聚合的业务域列表`
- 工作项格式：`domain-name: 业务域描述`

### 版本号规则
- 新增模块 → major + 1
- 修改现有模块 → minor + 1
- 文档修正 → patch + 1

### 变更标记
- `<!-- UPDATED: 日期 -->` - 标记更新的章节
- `<!-- BREAKING: 说明 -->` - 标记破坏性变更（如模块拆分）
- `<!-- TODO: 说明 -->` - 标记待下一阶段处理的事项

### Status 字段说明
在文档中添加注释，告诉用户如何修改 status：


## skill要解决的问题

1. 如何从 5 万字 PRD 中提取关键信息？
   - 使用 Read 工具分段读取大文档
   - 优先读取目录结构，识别章节
2. 如何处理 PRD 中的图片和文档引用？
   - 使用正则匹配 `![](...)` 和 `[](...)` 语法
   - 使用 Read 工具读取引用的图片（支持 PNG/JPG）和文档
   - 将图片内容和引用文档纳入分析上下文
3. 如何按领域划分模块？
   - 基于 DDD 原则识别业务域
   - 使用 AskUserQuestion 与用户确认划分方案
4. 如何做增量更新（对比旧版本，识别变更）？
   - 读取旧版本文档，对比差异
   - 只更新受影响的章节
5. 如何确保模块清单格式统一？
   - 严格按照 `module-name: 描述` 格式输出
6. 如何确保 frontmatter 字段完整？
   - 使用模板验证必需字段
7. 如何确保与用户交互？
   - 必须使用 EnterPlanMode 进入计划模式
   - 使用 AskUserQuestion 确认关键决策
   - 使用 ExitPlanMode 退出并执行
