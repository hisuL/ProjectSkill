# 系统边界

## 1. 目标

本文件统一描述当前系统在架构层的边界定义，避免“业务域”和“服务边界”分散在两份文档中产生重复或冲突。

本文件需要同时回答：
- 当前系统做什么，不做什么
- 当前系统有哪些业务域
- 业务域与后端服务如何映射
- 每个服务负责什么、不负责什么、拥有哪些主数据
- 服务之间通过什么稳定边界协作

## 2. 业务目标

系统需要让用户在单份 PDF 范围内完成：
- 上传文档
- 等待解析完成
- 基于该文档发起问答
- 查看引用并跳转到对应页
- 获取摘要、推荐问题并提交反馈

## 3. 范围边界

### 3.1 本轮范围内

- 单文档上传与处理
- 文档就绪状态查询
- 单文档问答
- 引用跳页
- 摘要与推荐问题
- 反馈收集

### 3.2 本轮范围外

- 多文档知识库
- OCR
- 权限与租户
- Office 文档支持
- Rerank 和复杂检索增强
- 内容安全平台化能力

## 4. 业务域与服务映射

当前架构层只保留两个粗粒度业务域，并与后端对外服务保持一一映射：

| 业务域 | 对应服务 | 说明 |
| --- | --- | --- |
| 文档处理域 | `document-ingestion` | 负责文档从上传到进入可检索状态前的全链路处理 |
| 对话编排域 | `conversation-orchestrator` | 负责问答、引用装配、会话、摘要、推荐问题和反馈 |

采用一一映射的原因：
- 当前系统规模较小，业务域和服务边界没有拆开的必要
- 主数据归属清晰，便于后续后端详细设计按服务展开
- 可以减少架构文档之间的重复定义

前端在当前阶段只作为外部调用方存在，不单独定义为架构层业务域。

## 5. 服务边界

### 5.1 `document-ingestion`

职责：
- 接收上传请求
- 存储原始 PDF
- 解析文档内容
- 进行切片和向量化
- 管理文档处理状态和失败原因
- 输出可供检索的 chunk 与引用定位元数据
- 提供预览所需元数据
- 支持取消处理中任务

不负责：
- 问答生成
- 会话管理
- 推荐问题编排
- 反馈记录

拥有的主数据：
- 原始 PDF
- 文档处理状态
- 解析产物
- chunk 元数据
- embeddings

对外边界：
- 面向前端提供上传、状态查询和预览相关能力
- 面向 `conversation-orchestrator` 提供 `doc_id` 范围内可检索数据边界

### 5.2 `conversation-orchestrator`

职责：
- 接收问题和最近多轮上下文
- 在当前 `doc_id` 范围内完成检索
- 组织上下文并调用模型生成回答
- 装配引用锚点
- 管理会话
- 返回摘要和推荐问题
- 接收反馈

不负责：
- 原始文档存储
- 文档解析
- 切片和 embedding 生成
- 文档处理状态主数据维护

拥有的主数据：
- 会话历史
- 反馈记录
- 摘要结果
- 推荐问题

对外边界：
- 面向前端提供问答、会话、反馈能力
- 面向 `document-ingestion` 消费只读检索数据

## 6. 服务之间的最小协作面

### 6.1 `document-ingestion -> conversation-orchestrator`

输出边界：
- 只输出当前 `doc_id` 下可检索的 chunk、embedding 和引用定位元数据
- 只输出稳定状态，不暴露解析实现细节

约束：
- `conversation-orchestrator` 只能消费 `ready` 状态文档
- 不允许跨 `doc_id` 检索

### 6.2 `conversation-orchestrator -> document-ingestion`

反向边界：
- 不反向修改文档处理主数据
- 不负责切片策略、向量化策略和文件存储策略

## 7. 划分原则

本轮后端微服务按 API 能力边界划分，而不是按内部进程或 worker 形态划分。

判断原则：
- 是否对外提供独立业务能力
- 是否拥有独立主数据
- 是否能形成清晰的 API 契约边界
- 是否值得单独演进和交付

## 8. 运行形态说明

- `document-ingestion` 内部可以有同步 API 面和异步任务执行面
- `conversation-orchestrator` 内部可以有流式响应和异步摘要生成任务

这些都属于服务内部实现，不在本阶段继续细分为独立微服务。
